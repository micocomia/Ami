# Fairness Validator Module for Learner Profiling

## Context

The AI Tutor system creates and updates learner profiles using an LLM. The profile includes FSLSM learning style dimensions, SOLO-taxonomy proficiency levels, and behavioral patterns — all inferred from resume text, persona selection, and interaction data. Currently there are **zero checks** for whether these inferences are fair and evidence-based. This module adds a post-processing fairness validation layer that reviews learner profiles for unjustified assumptions.

**Branch:** `sprint-2-ethics-bias-enhancement`

**Non-overlapping with:** The existing Bias Auditor (which audits *skill gap assessments* only). This module audits the *learner profile* pipeline — a completely separate data structure and code path.

## What It Does

1. **FSLSM inference validation** (LLM) — Flags if FSLSM dimensions deviate from persona baseline without evidence from the resume (e.g., don't assume "Engineer = Sensing" without supporting proof)
2. **SOLO justification check** (LLM) — Warns if proficiency level assessments in cognitive status lack justification from learner information
3. **Confidence-to-evidence alignment** (deterministic) — Checks that extreme FSLSM values (strong preferences) have actual information to support them rather than being assumed from job title or field
4. **Persona stereotype detection** (deterministic) — Validates that persona descriptions and additional_notes fields don't contain stereotypical language tied to demographics

## Architecture

Strictly a **post-processing layer** — takes an existing learner profile + learner information as input, returns a fairness validation report. Zero modifications to the existing profile creation/update pipeline.

```
POST /create-learner-profile-with-info  (existing, unchanged)
  -> returns learner_profile

POST /validate-profile-fairness  (NEW)
  -> takes learner_profile + learner_information + persona_name
  -> returns fairness validation report
```

## Implementation Steps

### Step 1: Schemas (`backend/modules/learner_profiler/schemas.py`)

Add at the end of the file:

- **`FairnessCategory`** enum: `fslsm_unjustified_deviation`, `solo_missing_justification`, `confidence_without_evidence`, `stereotypical_language`
- **`FairnessSeverity`** enum: `low`, `medium`, `high`
- **`FairnessFlag`** model: `field_name` (which profile field is flagged), `fairness_category`, `severity`, `explanation` (max 40 words), `suggestion` (max 30 words)
- **`FSLSMDeviationFlag`** model: `dimension` (e.g. "fslsm_processing"), `persona_value` (float), `profile_value` (float), `deviation` (float), `issue` (str)
- **`ProfileFairnessResult`** model:
  - `fairness_flags: List[FairnessFlag]` (from LLM)
  - `fslsm_deviation_flags: List[FSLSMDeviationFlag]` (deterministic)
  - `overall_fairness_risk: FairnessSeverity` (low/medium/high)
  - `checked_fields_count: int`
  - `flagged_fields_count: int`
  - `ethical_disclaimer: str` (default: "This learner profile was generated by AI based on limited information. Learning style preferences and proficiency levels are estimates and may not fully represent your actual preferences or abilities.")

### Step 2: Prompts (`backend/modules/learner_profiler/prompts/fairness_validator.py`)

- **System prompt**: Defines the Fairness Validator role with directives to:
  - Check if FSLSM dimension values are justified by evidence in the learner's resume/background, or if they appear to be inferred from job title, field of study, or demographic cues alone
  - Verify that each proficiency level in `cognitive_status` (both mastered and in-progress skills) has justification from the learner information — flag any that appear assumed without evidence
  - Detect stereotypical language in `additional_notes`, `behavioral_patterns`, or `learner_information` summary fields (e.g., assumptions like "as an engineer, prefers hands-on learning")
  - Use the 4 fairness categories for classification
  - Be calibrated, not alarmist — empty flags list is valid
  - Assign `overall_fairness_risk` based on flag count and severity
  - NOT re-generate the profile — only validate fairness of the existing one
- **Task prompt**: Takes `{learner_information}`, `{learner_profile}`, and `{persona_name}` as placeholders
- Update `prompts/__init__.py` to export

### Step 3: Agent (`backend/modules/learner_profiler/agents/fairness_validator.py`)

- **`FairnessValidationPayload`**: Pydantic input model with `learner_information` (str), `learner_profile` (dict), `persona_name` (str, optional)
- **`FairnessValidator(BaseAgent)`**:
  - Constructor: `super().__init__(model, system_prompt, jsonalize_output=True)` — no tools needed
  - `validate_profile(input_dict)` method:
    1. Validate input with `FairnessValidationPayload`
    2. Serialize `learner_profile` to JSON string for the prompt
    3. Call `self.invoke()` — LLM returns `fairness_flags` + `overall_fairness_risk`
    4. Run deterministic `_check_fslsm_deviation()`:
       - Compares each FSLSM dimension in the profile against the persona baseline
       - Flags deviations > 0.4 from persona baseline (large unexplained shifts)
       - Only runs if `persona_name` is provided and maps to known persona values
    5. Run deterministic `_check_stereotype_keywords()`:
       - Scans `additional_notes`, `behavioral_patterns.additional_notes`, and `learner_information` summary for stereotype phrases
       - Keyword list: "as an engineer", "typical for", "as expected from", "naturally inclined", "inherently", "as a woman", "as a man", "given their age", "from that background", "people from"
       - Flags matches as `stereotypical_language` category
    6. Merge LLM output with deterministic flags
    7. Compute `checked_fields_count` (FSLSM dims + cognitive skills + behavioral fields) and `flagged_fields_count`
    8. Promote `overall_fairness_risk` to "medium" if deterministic flags exist but LLM said "low"
    9. Validate with `ProfileFairnessResult` Pydantic model, return `model_dump()`
- **`validate_profile_fairness_with_llm()`**: Top-level convenience function
- Update `agents/__init__.py` and `__init__.py` to export

### Step 4: API Layer

- **`api_schemas.py`**: Add `ProfileFairnessRequest(BaseRequest)` with:
  - `learner_profile: str`
  - `learner_information: str`
  - `persona_name: str = ""`
- **`main.py`**: Add endpoint after the existing `/create-learner-profile-with-info` block:

- Add new function:
  ```python
  def validate_profile_fairness(learner_profile, learner_information, persona_name="", llm_type=None):
      """Call the fairness validation endpoint and return the result."""
      cfg = get_app_config()
      llm_type = llm_type or cfg["default_llm_type"]

          "persona_name": persona_name,
          "llm_type": str(llm_type),

      return make_post_request(API_NAMES["validate_profile_fairness"], data)
  ```

|----------|-------------------|

## Key Design Decisions

1. **Separate from Bias Auditor**: This module lives in `learner_profiler/` (not `skill_gap/`) because it audits a different data structure (learner profile vs skill gaps). No shared code or cross-dependencies.

2. **Persona-aware validation**: The FSLSM deviation check requires knowing which persona the user selected. This is passed from the frontend, where `st.session_state["learner_persona"]` stores it.

3. **Deterministic + LLM hybrid**: Like the bias auditor, we use deterministic checks (FSLSM deviation, stereotype keywords) alongside LLM judgment (SOLO justification, evidence-based FSLSM inference). Deterministic checks are fast, reliable, and testable; LLM checks handle nuanced reasoning.

4. **Non-blocking**: If the validation endpoint fails, the user can still proceed. The ethical disclaimer is hardcoded as a fallback.
